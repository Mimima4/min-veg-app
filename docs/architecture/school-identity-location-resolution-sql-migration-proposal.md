## 4. Physical enum/check strategy proposal

Proposal only (not final decision):

- Prefer **constrained text + check constraints** as initial physical strategy.
- Keep native database enum as a considered alternative, not accepted yet.

Tradeoff:

- constrained text + checks: easier evolution and deprecation management;
- native enums: stronger DB-level typing, but less flexible evolution.

Boundary:

- final enum strategy requires explicit SQL/migration proposal acceptance.
- enum changes must never silently change runtime behavior.

## 5. Table-by-table SQL-shape proposal (non-executable)

### 5.1 `source_school_observations`

- Likely column categories: `uuid`, `text`, `timestamptz`, `jsonb`.
- Nullability: source identity/snapshot/profession/stage and observed timestamp required.
- PK concept: single surrogate identifier.
- FK concept: parent for candidate/decision tables.
- Unique concept: dedupe on source snapshot + source school identity context.
- Index concepts: by snapshot, county/profession/stage.
- Check concepts: non-empty source identifiers.
- Mutability: append-only.
- JSON strategy: optional observation payload with minimum shape contract.
- Security/RLS: read access scoped to service/operator classes (policy-level only).
- Forbidden contents: final publishability/resolution decisions.

### 5.2 `school_identity_candidates`

- Likely categories: `uuid`, `text`, `jsonb`, `timestamptz`.
- Nullability: observation reference, identity candidate key, signal version required.
- PK concept: surrogate row id.
- FK concept: to source observation.
- Unique concept: dedupe per observation/candidate/version.
- Index concepts: observation lookup, candidate identity lookup.
- Check concepts: non-empty candidate key and signal version.
- Mutability: append-only by generation.
- JSON strategy: structured `match_signals` contract required.
- Security/RLS: generated by system boundary; operator read-only in normal mode.
- Forbidden contents: direct publication approval.

### 5.3 `identity_aliases`

- Categories: `uuid`, `text`, `timestamptz`, `jsonb`.
- Nullability: identity key, alias label/type, evidence reference required.
- PK concept: surrogate id.
- FK concept: conceptual link to identity model key.
- Unique concept: normalized alias uniqueness per identity/type/language.
- Index concepts: by identity key and normalized alias.
- Check concepts: alias type must be approved enum-like value.
- Mutability: append with validity intervals/supersession.
- JSON strategy: evidence references with minimal structure.
- Security/RLS: service/operator only.
- Forbidden contents: runtime route publish flags.

### 5.4 `school_locations`

- Categories: `uuid`, `text`, `jsonb`, `timestamptz`.
- Nullability: location key and identity key required.
- PK concept: surrogate id.
- FK concept: conceptual identity + optional NSR institution link.
- Unique concept: identity/location uniqueness; optional identity+institution uniqueness.
- Index concepts: identity key, county/municipality, NSR institution.
- Check concepts: stable key format and evidence presence.
- Mutability: versioned/superseded, no destructive overwrite.
- JSON strategy: location evidence references.
- Security/RLS: service/operator scope.
- Forbidden contents: implicit publication states.

### 5.5 `school_identity_resolution_decisions`

- Categories: `uuid`, `text`, `jsonb`, `timestamptz`, `boolean` (optional active marker alternative not preferred).
- Nullability: decision state, actor type, basis version, reason codes, audit ref, decided timestamp required.
- PK concept: surrogate id.
- FK concept: observation id + conceptual links to identity/location/provider entities.
- Unique concept: one active decision per observation.
- Partial unique concept: active-row uniqueness via supersession marker (see section 6).
- Index concepts: active decisions, unresolved backlog, stale review queue.
- Check concepts: publish-oriented rows require audit+basis fields.
- Mutability: append-only with supersession.
- JSON strategy: reason/evidence payload contract required.
- Security/RLS: system + authorized operator actions.
- Forbidden contents: destructive overwrite; hidden manual overrides.

### 5.6 `programme_availability_publication_decisions`

- Categories: `uuid`, `text`, `jsonb`, `timestamptz`.
- Nullability: observation/programme/stage, publishability state, audit+basis fields required.
- PK concept: surrogate id.
- FK concept: to observation + conceptual linkage to active resolution decision.
- Unique concept: one active publication decision per observation/programme/stage.
- Partial unique concept: active-row uniqueness via supersession marker.
- Index concepts: publication-ready set, blocked-review set, by county/profession/stage through joins.
- Check concepts: publication gate prerequisites represented in state/field consistency checks.
- Mutability: append-only with supersession.
- JSON strategy: decision reason/evidence contract.
- Security/RLS: no public/family read; service/operator boundary.
- Forbidden contents: publishability without audit_ref + decision_basis_version.

### 5.7 `school_identity_review_events`

- Categories: `uuid`, `text`, `jsonb`, `timestamptz`.
- Nullability: target decision id, event type, actor id/type, timestamp required.
- PK concept: surrogate id.
- FK concept: target decision reference.
- Unique concept: event stream by id.
- Index concepts: backlog by event type/age, actor workload, unresolved queue.
- Check concepts: event type in approved set; payload must include reason/evidence linkage.
- Mutability: append-only.
- JSON strategy: event payload schema contract.
- Security/RLS: operator/service scope with strict write authority.
- Forbidden contents: replacing authoritative decision row.

## 6. Active-row uniqueness strategy

Canonical proposed mechanism:

- append-only rows with **`superseded_at IS NULL`** as active-row marker.

Rules:

- one active resolution decision per observation;
- one active publication decision per observation/programme/stage;
- supersession inserts new row and closes prior row with superseded marker;
- append-only history preserved.

Alternative:

- `is_active_current` is treated as a rejected alternative in this proposal unless separately approved.

Boundary:

- exact physical constraint implementation remains subject to SQL review.

## 7. JSON validation contracts

Minimum JSON shape contracts (proposal-level):

- `decision_reason_codes`: non-empty array of approved reason-code text.
- `match_signals`: object with signal keys + version metadata.
- `evidence_refs`: array/object of source references.
- `audit_payload`: object containing actor, before/after state, reason codes, evidence refs, snapshot, basis version, timestamp.
- `event_payload`: object scoped to review event with reason/evidence references.

Rules:

- malformed payloads must not qualify for publishable decision path.
- JSON contract versioning should be explicit in SQL/migration phase.

## 8. RLS/security and operator authority (policy-level)

Policy-level only (not final rules):

- define read boundaries for service/operator/system roles;
- system boundary can insert generated candidate rows;
- operator boundary can create review events and audited decision transitions;
- publishable approvals require audited authority;
- unresolved/review datasets are not public/family-facing.

Boundary:

- final RLS rules require separate security review.
- do not assume final auth/role topology beyond conceptual service/operator/system boundaries.

## 9. Stale severity / revalidation triggers

Hard-stale (blocks publication and requires revalidation):

- school code change;
- programme/stage availability change;
- source snapshot lineage drift;
- NSR institution active/private/location changes affecting semantics;
- alias/location evidence drift invalidating active basis.

Soft-stale (advisory/review signal):

- label/display variants with stable identity evidence;
- non-material metadata drift.

Interaction:

- `source_snapshot_label`, `signal_version`, and `decision_basis_version` must remain consistent with active decision.
- hard-stale requires supersession before publication.

## 10. Key collision and merge/split governance

- identity_key collision handling must be deterministic and audited.
- location_key collision handling must be deterministic and audited.
- identity merge flow must preserve lineage via supersession/mapping.
- identity split flow must preserve lineage via supersession/mapping.
- no destructive key reuse.
- no display-label-only key derivation.

## 11. Retention / archival and backfill sequencing

Retention/archival proposal:

- append-only history retained with supersession chain.
- no destructive delete while referenced by decision/audit rows.
- retention minimum is policy-owned (compliance owner).
- archival must preserve decision/audit chain.

Backfill boundary:

- no backfill execution approved here.
- any backfill algorithm requires separate ADR/plan and approval.
- this document defines sequencing principles only.

Rollback sequencing (proposal-level):

- rollback strategy must preserve audit/decision history and avoid silent truth loss.

## 12. Publication safety checks (pre-PSA integration)

Before any PSA write-path integration:

- active publication decision exists;
- `publishability_state = publishable`;
- active identity resolution decision exists;
- decision is not stale;
- required audit fields are present;
- state is not `unsupported_losa` / `external_delivery`;
- provider/location semantics are resolved.
